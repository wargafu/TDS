DSTD — Tchad Design System

Système typographique officiel

Documentation minimale et exemples visuels pour le système typographique DSTD. Ce guide couvre l’architecture des polices, l’échelle de taille, les styles textuels, le multilingue et les règles d’accessibilité.

1. Architecture des polices

DSTD utilise une architecture typographique claire et performante : Source Sans 3 pour le Latin, Noto Naskh Arabic pour l’Arabe, Public Sans comme secours institutionnel et JetBrains Mono pour les usages monospace.

html {
  font-family: "Source Sans 3", "Public Sans", system-ui, -apple-system, "Segoe UI", Roboto, "Helvetica Neue", Arial, sans-serif;
}

html[lang="ar"], :lang(ar) {
  direction: rtl;
  font-family: "Noto Naskh Arabic", "Noto Sans Arabic", system-ui, sans-serif;
}

code, kbd, pre {
  font-family: "JetBrains Mono", ui-monospace, SFMono-Regular, Menlo, Monaco, "Roboto Mono", "Courier New", monospace;
}

Cette pile est conçue pour la production institutionnelle : elle est cohérente, compatible cross-browser et prête pour l’usage web. Le fallback inclut les polices système standard pour réduire les risques de rendu.

2.1. Système d’espacement

DSTD applique une grille modulaire de 4px rendue en rem pour un écosystème stable et accessible. Chaque token représente un multiple de 4px : par exemple spacing-4 = 16px / 1rem et spacing-16 = 64px / 4rem. Cette échelle simple est pensée pour les portails citoyens, les tableaux de bord administratifs, les formulaires complexes et les pages multi-modules.

Principes d’usage

  • Small (4–12px) : micro-spacing, icônes, badges, légendes.
  • Medium (16–24px) : padding de champ, marges de composants, lignes de formulaire.
  • Large (32–56px) : sections, cards, modales, zones d’actions.
  • Extra-large (64–96px) : marges de page, separation entre grandes zones.
  • Éviter le « sur-spacing » sur mobile : ne pas dépasser 40px entre éléments fonctionnels dans une colonne étroite.
  • Garder la cohérence verticale/horizontale : une même valeur doit être réutilisée pour les margins et paddings apparentés.

Règles de layout

  • Sections : 24–32px sur mobile, 32–48px sur desktop.
  • Pages : 32px de padding interne mobile, 40–48px desktop; containers max-width 1024px ou 1200px.
  • Cards : 16–24px de padding interne, 16–24px de marge externe.
  • Modals : 24–32px de padding interne avec 24px de gap entre champs.
  • Tables : 16–24px de padding dans les cellules et 24px entre groupes de lignes.

Responsive et accessibilité

  • Sur mobile, privilégier 16–24px pour les blocs fonctionnels et 32px max pour les zones espacées.
  • Maintenir des cibles tactiles ≥ 44px en combinant spacing et dimensions des boutons.
  • Des espacements suffisants améliorent le focus clavier et la lisibilité des formulaires.
  • Éviter la densité excessive : ne pas réduire les gaps en dessous de 8px pour les éléments interactifs adjacents.

2.2. Rayons et ombres

Le système DSTD des arrondis reste sobre et institutionnel. Les rayons servent à hiérarchiser les surfaces sans verser dans un style trop « startup ».

TokenValeurUsage recommandé
radius-none0 / 0remTables, cards formelles, composants extrêmement structurés
radius-xs2px / 0.125remSurfaces neutres, containers de données, cards gouvernementales
radius-sm4px / 0.25remBoutons, inputs, tags, barres d’actions
radius-md8px / 0.5remCards, modals, menus, sections de formulaire
radius-lg12px / 0.75remDialogues, overlays, panels plus doux
radius-xl16px / 1remSections majeures, encarts mis en avant
radius-full9999pxBadges, tags, pills, boutons full-round

Règles DSTD pour les arrondis

  • Faible : `none`, `xs` et `sm` pour les interfaces formelles et les données.
  • Moyen : `md` pour les cartes, panels et modales afin de conserver une densité lisible.
  • Élevé : `lg` et `xl` seulement sur des surfaces isolées ou des sections d’accueil.
  • Full : réservé aux éléments pétillants de statut ou aux badges, pas aux surfaces principales.
  • Éviter les coins trop arrondis sur les tables, cartes de services et sections de données.

Correspondance composants

  • Buttons : `sm`
  • Inputs / forms : `sm`
  • Cards : `xs` ou `md` selon le niveau de formalité
  • Modals : `md`
  • Tables : `none` ou `xs`
  • Badges / tags : `full` ou `sm`
  • Navigation / menus : `sm`

Accessibilité et cohérence visuelle

  • Les arrondis ne doivent pas masquer la structure des composants ni diminuer la lisibilité.
  • Les focus states doivent rester nets, avec un contraste de bordure suffisant.
  • La même valeur de radius doit être appliquée à des composants similaires pour éviter les incohérences.
  • Le radius est fixe sur mobile et desktop pour garantir une identité stable.

Guide d’utilisation des ombres

  • Les ombres servent uniquement à la hiérarchie visuelle, pas à la décoration.
  • Privilégier shadow-none pour les tables, listes de données et surfaces très formelles.
  • Utiliser shadow-xs ou shadow-sm pour des composants interactifs légers comme les dropdowns et boutons hover.
  • Réserver shadow-md à la profondeur standard des modals et des panels.
  • shadow-lg et shadow-xl doivent être appliqués uniquement aux overlays et aux éléments flottants critiques.
  • Éviter d’ajouter une ombre sur des composants déjà très espacés ou colorés ; un bord/netteté suffit souvent.
  • En dark mode, baisser l’opacité des ombres pour conserver une sensation discrète et éviter l’effet brouillé.

2.3. Motion et z-index

Le système de motion DSTD est conçu pour être sobre, fonctionnel et accessible. Les animations doivent améliorer la compréhension de l’interface sans devenir décoratives.

EasingVariable CSSValeurUsage

linear--dstd-easing-linearlinearCas rares : loaders, progress bar

Philosophie motion DSTD

  • Motion discrète, au service de la hiérarchie et de la compréhension.
  • Animations uniquement pour indiquer un changement d’état ou une relation spatiale.
  • Aucune animation décorative, aucun « bounce » inutile.
  • Les mouvements rapides sont réservés aux interactions tactiles mineures.

Accessibilité

  • Supporter prefers-reduced-motion pour désactiver les animations.
  • Les animations ne doivent jamais être nécessaires pour comprendre le flux.
  • En cas de mode réduit, utiliser instant ou supprimer les transitions.
  • Les focus states doivent rester visibles et ne pas dépendre d’animations.

Usage par composant

  • Buttons : hover/focus avec fast + standard.
  • Dropdowns : entrée/sortie avec normal + enter/exit.
  • Modals : ouverture avec slow + enter, fermeture avec exit.
  • Tooltips : apparition discrète fast + enter.
  • Toasts : affichage avec slow, disparition avec exit.
  • Navigation : transitions de page ou panneaux avec normal + standard.

Z-index DSTD

La hiérarchie z-index DSTD est verrouillée et minimale. Les valeurs sont espacées de 100 pour conserver une marge de manœuvre, tout en restant explicites et sans ambiguïté.

TokenVariable CSSValeurUsage
base--dstd-z-base0Contenu de base, layout normal
dropdown--dstd-z-dropdown100Menus déroulants, popovers légers
sticky--dstd-z-sticky200Barres sticky, navigation fixe
overlay--dstd-z-overlay300Overlays masquants et backdrops
modal--dstd-z-modal400Fenêtres modales
toast--dstd-z-toast500Notifications, toasts
tooltip--dstd-z-tooltip600Tooltips, info-bulles critiques

Règles strictes

  • Interdiction des z-index arbitraires hors de l’échelle DSTD.
  • Modal > dropdown toujours.
  • Tooltip > modal uniquement lorsque le tooltip est lié à une interaction modale et est absolument positionné dans le même stacking context.
  • Sticky ne doit jamais recouvrir une modal ou un overlay.
  • Overlay bloque l’interaction avec la page sous-jacente.
  • Ne pas utiliser de valeurs intermédiaires sans justification design system.

Gestion des stacking contexts

  • Éviter de créer des stacking contexts inutiles avec transform, opacity, filter, will-change sur des éléments empilés.
  • Si un composant doit créer un stacking context, vérifier que sa valeur z-index utilise bien le token DSTD correspondant.
  • Les règles d’empilement s’appliquent dans le même context ; ne pas compter sur des valeurs z-index locales pour traverser plusieurs contexts.

Tableau de priorité absolue

  • tooltip (600) > toast (500)
  • toast (500) > modal (400)
  • modal (400) > overlay (300)
  • overlay (300) > sticky (200)
  • sticky (200) > dropdown (100)
  • dropdown (100) > base (0)

3. Boutons DSTD

Le système de boutons DSTD est conçu pour être direct, lisible et fiable dans les parcours administratifs. Les variantes sont limitées, hiérarchisées et compatibles HTML-first, sans dépendre de JavaScript.

Philosophie des boutons

  • Un bouton signifie une action claire ; il ne doit pas distraire.
  • La hiérarchie se manifeste par l’usage de variantes simples : primary, secondary, tertiary, danger, success.
  • Les styles sont pensés pour les formulaires publics : robustes, contrastés, accessibles.
  • La palette reste institutionnelle et sobre, sans couleurs vives non nécessaires.

4. Inputs DSTD

Le système de champs DSTD s’appuie sur la même sobriété que les boutons : structure claire, états visibles et variantes limitées pour éviter les erreurs dans les formulaires publics.

Structure standard

  • Label obligatoire associé par for/id.
  • Champ : input, textarea, select ou contrôle booléen.
  • Helper text optionnel pour précision.
  • Message d’erreur visible et prioritaire.
  • Hint accessible via aria-describedby.

5. Multilingue (français / arabe)

Le système est conçu pour gérer automatiquement le contenu français en Latin et l’arabe en RTL avec une police dédiée.

html {
  font-family: "Source Sans 3", "Public Sans", system-ui, -apple-system, "Segoe UI", Roboto, Arial, sans-serif;
}

html[lang="ar"], :lang(ar) {
  direction: rtl;
  font-family: "Noto Naskh Arabic", "Noto Sans Arabic", system-ui, sans-serif;
  line-height: 1.6;
}

6. Accessibilité

DSTD applique les bonnes pratiques WCAG 2.1 AA :

  • Texte de base à 16px minimum.
  • Line-height généreuses (1.4–1.5 pour le corps).
  • Longueur de ligne cible 45–75 caractères.
  • Zoom 200% : la mise en page reste utilisable.
  • Les accents d’alerte jaunes utilisent du texte ou des icônes foncés pour maintenir un contraste suffisant.

Les petites tailles (12–14px) sont réservées aux légendes ou aides non critiques.

Typographie responsive

DSTD recommande d’adapter les titres et les échelles de texte entre mobile, tablette et desktop sans casser la hiérarchie.

h1 {
  font-size: clamp(2rem, 5vw, 3rem);
  line-height: 1.12;
}

@media (min-width: 768px) {
  h1 {
    font-size: clamp(2.5rem, 4vw, 3.5rem);
  }
}

8. Notes de déploiement

Pour un déploiement stable, self-hostez les fichiers WOFF2 de Source Sans 3, Noto Naskh Arabic, Public Sans et JetBrains Mono. Utilisez le mode font-display: swap et des subsets par langue pour optimiser la performance.
