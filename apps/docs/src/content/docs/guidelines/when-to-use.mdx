# Quand utiliser DSTD

DSTD est conçu pour des contextes spécifiques. Cette page aide les équipes à décider
si DSTD est le bon choix pour leur projet.

---

## DSTD est fait pour

### Services numériques gouvernementaux
Portails citoyens, applications de démarches administratives, formulaires de demande de documents,
tableaux de bord ministériels, sites de publication de l'État.

**Exemples concrets :**
- Portail de demande de carte d'identité nationale
- Interface de déclaration de revenus en ligne
- Tableau de bord de suivi des dossiers pour agents administratifs
- Site officiel d'un ministère ou d'une préfecture
- Application de paiement des taxes locales

### Services para-étatiques
Organismes publics autonomes, entreprises publiques, institutions sous tutelle de l'État,
projets financés par l'État du Tchad.

### Projets nécessitant cohérence institutionnelle
Tout projet où l'identité visuelle de l'État est un prérequis, ou où les citoyens
doivent reconnaître immédiatement qu'ils interagissent avec un service officiel.

---

## DSTD n'est pas fait pour

### Applications commerciales privées
DSTD n'est pas un framework CSS généraliste. Son esthétique institutionnelle
ne convient pas aux applications commerciales, fintech, e-commerce, etc.

### Startups et produits marketing-first
Les interfaces à fort impact visuel, landing pages avec animations, produits SaaS
nécessitent des librairies plus flexibles (Tailwind + Headless UI, shadcn/ui, etc.)

### Applications très spécialisées
Outils de data visualisation avancés, éditeurs de contenu complexes, applications temps-réel
nécessitent des composants qui dépassent le périmètre actuel de DSTD.

<Aside type="note" title="DSTD comme couche de base">
  Même dans des projets commerciaux, les tokens DSTD (couleurs, espacement)
  peuvent servir de point de départ pour maintenir la cohérence visuelle
  avec les services gouvernementaux partenaires.
</Aside>

## Critères d'évaluation

Avant d'adopter DSTD, répondez à ces questions :

| Question | Si oui | Si non |
|---|---|---|
| Mon service est-il destiné aux citoyens ou agents de l'État ? | ✅ DSTD recommandé | Évaluer selon le contexte |
| L'identité visuelle de l'État est-elle requise ? | ✅ DSTD recommandé | Autre framework possible |
| L'accessibilité WCAG AA est-elle un prérequis ? | ✅ DSTD conforme | Tout framework accessible |
| Le projet supporte-t-il HTML-first / no-JS ? | ✅ DSTD compatible | Évaluer les dépendances |
| Mon interface a-t-elle besoin d'animations avancées ? | ⚠️ Hors périmètre DSTD | — |

---

## Processus d'adoption recommandé

1. **Évaluation** — Identifier les interfaces concernées par DSTD
2. **Audit existant** — Inventorier les composants et styles actuels
3. **Migration des tokens** — Remplacer les couleurs/spaces custom par les tokens DSTD
4. **Migration composants** — Adopter les classes CSS DSTD progressivement
5. **Validation accessibilité** — Tester avec axe-core et navigation clavier
6. **Formation équipe** — Partager ce guide de documentation avec les développeurs
